Runway Remote Agent

Architecture, protocol, deployment, and security reference for the Runway agent system.

Overview

The Runway Remote Agent is a persistent Rust binary that runs on Linux servers as a systemd service. It receives code deployments from the Runway macOS app (or CLI) over gRPC, manages execution history in a local SQLite database, runs scheduled jobs independently, and supports remote self-upgrade.

Key principle: Zero runtime dependencies + full persistence. The agent is a single static binary with embedded SQLite. All execution history, logs, events, and schedules survive restarts. The app polls the agent for events when connected — no direct network connectivity from agent to app required.

Message Definitions

ExecuteRequest

message ExecuteRequest {
  string project_id  = 1;  // Unique project identifier
  string runtime     = 2;  // "python", "node", "go", "shell", "rust"
  string entrypoint  = 3;  // Filename: "main.py", "index.js", "run.sh"
  bytes  code        = 4;  // Inline code content (the file bytes)
  string working_dir = 5;  // Fallback working directory (if no inline code)
}

ExecuteResponse (streamed)

message ExecuteResponse {
  string stream    = 1;  // "stdout" or "stderr"
  string text      = 2;  // One line of output
  string timestamp = 3;  // RFC 3339 timestamp
}

HealthResponse

message HealthResponse {
  string agent_version  = 1;  // e.g. "0.1.0"
  string status         = 2;  // "healthy"
  uint64 uptime_seconds = 3;  // Seconds since agent started
  string os             = 6;  // e.g. "linux" (from std::env::consts::OS)
  string arch           = 7;  // e.g. "x86_64" (from std::env::consts::ARCH)
}

StatusResponse

message StatusResponse {
  string agent_id             = 1;  // Hostname of the server
  string status               = 2;  // "running"
  double cpu_usage            = 3;  // Global CPU % (via sysinfo)
  uint64 memory_bytes         = 4;  // Used memory in bytes
  repeated ProjectStatus projects = 5;
}

Data Flow

Remote Execution Flow

When a user clicks Run with a remote target selected in the UI:

1
Read code from disk — Tauri command reads the entrypoint file from ~/Library/Application Support/com.runway.app/projects/{name}/{entrypoint} on macOS.
2
Look up target — Target's gRPC endpoint is retrieved from SQLite: http://{host}:{port}.
3
Connect via gRPC — AgentClient::connect(endpoint) establishes an HTTP/2 channel to the remote agent.
4
Send ExecuteRequest — The code bytes, runtime type, and entrypoint filename are sent in a single protobuf message.
5
Agent writes to deploy dir — Code is written to ~/.runway/deploys/{project_id}/v{n}/. A versioned directory is created per deployment, persisted in the deployments table.
6
Agent spawns process — Creates an executions row, then runs the command. Every log line is persisted to execution_logs AND streamed over gRPC simultaneously.
7
Stream logs back — stdout/stderr are captured line by line and sent as ExecuteResponse messages over the gRPC stream. Each line is also written to SQLite for later replay via GetExecutionLogs.
8
Tauri emits events — Each log line is emitted as a project-log Tauri event. The Terminal component renders them identically to local logs.
9
Completion — When the stream ends, a project-status-change event is emitted (idle or failed), and the run is recorded in SQLite. On the agent side, the executions row is updated with exit code + finished_at, and an execution_completed event is inserted into the events queue.

Deployment

Method 1: Build from Source

For development and testing. Requires Rust toolchain and protoc on the target server.

# On the Linux server:
sudo apt install -y protobuf-compiler build-essential
curl --proto '=https' --tlsv1.2 -sSf https://sh.rustup.rs | sh -s -- -y
source ~/.cargo/env

# Copy source and build
rsync -avz --exclude target --exclude node_modules your-mac:~/runway/ ~/runway/
cd ~/runway
cargo build -p runway-agent --release

# Run (bind to all interfaces for network access)
./target/release/runway-agent --listen-all --port 50051

Method 2: Install Script

For production servers. Downloads a pre-built binary and installs as a systemd service.

# Install
curl -sSL https://get.runway.dev | sudo bash

# Uninstall
curl -sSL https://get.runway.dev | sudo bash -s -- --uninstall

The install script (scripts/install-agent.sh) performs:

StepAction
1. OS DetectionLinux only (rejects macOS)
2. Arch Detectionx86_64 or aarch64
3. DownloadBinary from release URL to /usr/local/bin/runway-agent
4. User CreationCreates runway system user (no home, nologin shell)
5. systemd ServiceCreates unit file, enables, starts

systemd Service Configuration

[Unit]
Description=Runway Agent
After=network.target

[Service]
Type=simple
User=runway
ExecStart=/usr/local/bin/runway-agent --port 50051
Restart=always
RestartSec=5
Environment=RUST_LOG=info

[Install]
WantedBy=multi-user.target

Post-Install Commands

# Check status
systemctl status runway-agent

# View logs
journalctl -u runway-agent -f

# Restart
sudo systemctl restart runway-agent

Method 3: Register in Runway App

After the agent is running, register it as a target:

# Via CLI
runway targets add my-server --host 192.168.1.222 --port 50051
runway targets ping my-server

# Via UI
# Targets page → + Add Target → fill name/host/port → Add → Ping

Credential Storage

The credentials module (crates/runway-core/src/credentials.rs) stores secrets in the macOS Keychain:

FunctionKeychain Key PatternPurpose
store_ssh_key(target, key)target:{name}:ssh-keySSH private key for remote access
store_aws_credentials(profile, ak, sk)aws:{profile}:access-keyAWS credentials for Lambda targets
store_credential(key, value)Custom keyGeneric secret storage

All credentials use the macOS security-framework crate, which stores secrets in the system Keychain — encrypted at rest, protected by the user's login password.

Security Recommendations

  • LAN only: Without mTLS, only deploy agents on trusted networks.
  • SSH tunnel: For internet-facing agents, use ssh -L 50051:localhost:50051 user@server and connect to localhost:50051.
  • Firewall: Restrict port 50051 to known IPs via ufw or security groups.
  • Agent user: The install script creates a runway system user with no shell — processes run without root privileges.
  • Temp directory cleanup: Executed code lives in /tmp/runway-agent-{uuid}/ directories. These should be periodically cleaned.

The agent uses a HashMap<String, RunningChild> protected by a tokio::sync::Mutex to track running processes:

    abort_handle: tokio::task::AbortHandle,

Resource Monitoring

The Status RPC reports real system metrics via the sysinfo crate:

  • CPU: Global CPU usage percentage
  • Memory: Used memory in bytes
  • Hostname: System hostname (used as agent_id)
  • Running projects: List of active project_ids with start times

Client Library

The gRPC client (crates/runway-core/src/agent_client.rs) is used by both the Tauri app and the CLI:

use runway_core::agent_client::AgentClient;

// Connect
let mut client = AgentClient::connect("http://192.168.1.222:50051").await?;

// Health check
let health = client.health().await?;
println!("v{}, uptime {}s", health.version, health.uptime_seconds);

// Execute code remotely
let code = std::fs::read("main.py")?;
let logs = client.execute(
    "my-project",    // project_id
    "python",        // runtime
    "main.py",       // entrypoint
    "/tmp",          // working_dir
    Some(&code),     // inline code bytes
).await?;

for line in &logs {
    println!("[{}] {}", line.stream, line.text);
}

// Stop
client.stop("my-project").await?;

// System status
let status = client.status().await?;
println!("CPU: {:.1}%, Memory: {}MB", status.cpu_usage, status.memory_bytes / 1024 / 1024);

Current Limitations

LimitationImpactWorkaround
No TLS in transit Code and logs are transmitted in plaintext Use on trusted LAN or SSH tunnel. mTLS infra built but not wired.
No authentication Anyone who can reach port 50051 can execute code Firewall rules, bind to localhost + SSH tunnel
No execution timeout Processes can run indefinitely on the agent Use Stop command or process supervisor
No container sandboxing Code runs as the agent's OS user, no isolation Run agent as a dedicated low-privilege user

Resolved in persistent agent redesign:

  • Multi-file deploy — Deploy RPC accepts tarballs, extracts to persistent directory
  • Dependency install — Deploy runs pip install, npm install, go mod download automatically
  • Temp cleanup — Deployments go to ~/.runway/deploys/, old versions auto-pruned (keep last 5)
  • Log persistence — All execution logs stored in SQLite, replayable via GetExecutionLogs
  • Offline scheduling — Agent runs cron jobs independently, events queued for app to poll
